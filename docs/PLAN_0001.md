# План реализации RFC-0001

Источник: `docs/rfcs/0001.md` (`review`).

## Допущения

- Реализация будет оформлена как долгоживущий демон `ssh-key-sync`.
- Проект реализуется как Rust CLI/daemon; сборка, тесты, форматирование и lint запускаются через цели `Makefile`.
- `SID` — публичный идентификатор группы синхронизации.
- `SID_TOKEN` — обязательный секрет группы; он не логируется, не передается в открытом виде и используется только для HMAC-подписи сообщений.
- Первая версия только добавляет и обновляет управляемый блок ключей в `authorized_keys`; автоматическое удаление ключей исчезнувших участников не входит в scope.

## Правила реализации из `AGENTS.md`

- Перед кодированием явно фиксировать допущения; если есть несколько трактовок, не выбирать молча.
- Делать минимальную реализацию RFC-0001 без speculative features и лишних абстракций.
- Изменять только файлы, необходимые для текущей фазы; не делать побочные refactor/cleanup.
- Для каждой фазы держать проверяемые критерии готовности и завершать фазу только после соответствующих проверок.
- Хранить долгоживущий проектный контекст через `mem-cli`.

## Правила выполнения команд из `.github/copilot-instructions.md`

- Shell-команды запускать через `rtk`-прокси (например, `rtk cargo test`, `rtk git status`).
- Для метрик экономии токенов использовать команды `rtk gain`, `rtk gain --history`, `rtk discover`.

## Критерии готовности

- Демон стартует только при валидных `SID`, `SID_TOKEN`, локальном публичном ключе и доступном `authorized_keys`.
- Участники с одинаковым `SID` и корректным `SID_TOKEN` обмениваются публичными SSH-ключами через HTTP API.
- UDP announcement ускоряет обнаружение новых участников, а polling сохраняет eventual consistency.
- Сообщения с чужим `SID`, отсутствующей подписью, неверной HMAC-подписью, устаревшим timestamp или повторным nonce игнорируются.
- `authorized_keys` обновляется идемпотентно только внутри управляемого блока `# ssh-key-sync begin/end`.
- Недоступные или не прошедшие проверку участники не останавливают демон, но видны в логах и состоянии.
- Локальные критичные ошибки завершают демон с ненулевым кодом.
- Финальная проверка проходит через `make fmt`, `make lint`, `make test`, `make build`.

Проверка:

## Фаза 1. Конфигурация и идентичность участника

Задачи:

1. Описать конфигурацию демона:
   - `SID`;
   - `SID_TOKEN`;
   - локальный идентификатор участника;
   - HTTP listen address/port;
   - UDP announcement address/port;
   - `bootstrap-peers`;
   - sync interval, по умолчанию `30s + jitter`;
   - путь к публичному SSH-ключу;
   - путь к `authorized_keys`;
   - режим `dry-run` для миграционной проверки.
2. Поддержать чтение значений из аргументов запуска, переменных окружения и конфигурационного файла.
3. Запретить значения по умолчанию для `SID` и `SID_TOKEN`.
4. Исключить `SID_TOKEN` из логов, debug output и сообщений об ошибках.

Проверка:

1. Тесты на успешную загрузку конфигурации.
2. Тесты на ошибки при отсутствующих `SID` и `SID_TOKEN`.
3. Тесты на маскирование `SID_TOKEN` в диагностике.

## Фаза 2. HMAC-аутентификация сообщений

Задачи:

1. Реализовать канонический формат подписываемых данных: `SID`, sender id, timestamp, nonce, method/path или message type, body hash.
2. Реализовать подпись HMAC-SHA256 через `SID_TOKEN`.
3. Реализовать проверку подписи для HTTP-запросов, HTTP-ответов и UDP announcement.
4. Добавить защиту от replay:
   - допустимое окно timestamp;
   - краткоживущий cache использованных nonce.
5. Игнорировать сообщения с чужим `SID`, некорректной подписью, устаревшим timestamp или повторным nonce.

Проверка:

1. Unit-тесты на стабильность канонического формата.
2. Unit-тесты на валидные и невалидные HMAC-подписи.
3. Unit-тесты на replay через timestamp и nonce.

## Фаза 6. Цикл демона и обработка ошибок

Задачи:

1. Реализовать стартовую синхронизацию.
2. Реализовать фоновый sync loop с интервалом `30s + jitter`.
3. Совместить polling и событие от announcement.
4. Добавить graceful shutdown: остановить новые циклы, дождаться текущей безопасной операции записи, завершиться с понятным кодом.
5. Разделить ошибки:
   - локальные критичные ошибки завершают демон;
   - ошибки отдельных участников логируются и не останавливают демон.
6. Добавить состояние демона: последний успешный sync, последние ошибки peers, количество известных участников.

Проверка:

1. Unit-тесты классификации ошибок.
2. Интеграционный тест: недоступный peer не останавливает демон.
3. Интеграционный тест: локальная ошибка `authorized_keys` завершает демон.
4. Тест graceful shutdown во время sync cycle.

## Риски и открытые уточнения

- UDP broadcast может быть недоступен или ограничен в некоторых container networks; polling через `bootstrap-peers` должен оставаться надежным fallback.
- HMAC с общим `SID_TOKEN` проще для первой версии, но не дает индивидуальной идентичности каждого участника; mTLS или pinning можно рассмотреть отдельным RFC.
- Автоматическое удаление ключей исчезнувших участников намеренно отложено, чтобы не ломать доступ из-за временных сетевых проблем.
- Нужно заранее определить допустимое окно timestamp и размер nonce cache, чтобы replay-защита не ломалась при небольшом рассинхроне времени.
