## Problem

Есть несколько хостов или контейнеров, на которых запускается один и тот же инструмент.
После старта эти хосты должны иметь возможность подключаться друг к другу по SSH.

Сейчас для этого требуется заранее вручную собрать публичные SSH-ключи всех участников и
разложить их по `authorized_keys` на каждом хосте. Такой подход неудобен для контейнеров и
динамических окружений: состав участников может меняться, контейнеры могут пересоздаваться, а
ручная синхронизация ключей легко приводит к ошибкам и рассинхронизации.

## Goal

Инструмент должен работать как демон и автоматически обмениваться SSH-публичными ключами между
известными хостами. После синхронизации каждый участник должен иметь возможность подключаться к
остальным по SSH, включая участников, которые появились уже после старта демона.

Цели:

- обнаруживать или получать список остальных участников кластера;
- отделять независимые группы синхронизации внутри одной сети через SID и SID_TOKEN;
- опубликовать собственный публичный SSH-ключ;
- получить публичные SSH-ключи других участников;
- обеспечить безопасный обмен ключами между доверенными участниками;
- добавить полученные ключи в локальный `authorized_keys`;
- продолжать синхронизацию при появлении новых участников;
- сделать повторную синхронизацию безопасной и идемпотентной.

Не цели на этом этапе:

- управление приватными SSH-ключами;
- замена полноценной системы управления секретами;
- реализация сложной политики доступа между пользователями;
- синхронизация произвольных файлов, кроме публичных SSH-ключей.

## Design

### Общая идея

Каждый хост запускает инструмент как демон синхронизации. При старте демон:

1. определяет собственный публичный SSH-ключ;
2. объявляет в сети, что появился новый участник;
3. находит других участников;
4. передает им свой публичный ключ;
5. получает их публичные ключи;
6. обновляет локальный файл `authorized_keys`.

После начальной синхронизации демон продолжает работать и повторяет discovery и обмен ключами,
чтобы новые контейнеры могли присоединяться без перезапуска уже запущенных участников.

После этого любой участник, чей публичный ключ был успешно получен и записан, может подключаться к
данному хосту по SSH.

### Участники

Участником считается хост или контейнер, на котором:

- запущен этот инструмент;
- настроен SID группы синхронизации;
- настроен SID_TOKEN для безопасного обмена внутри группы;
- доступен SSH-сервер;
- есть локальный публичный ключ, который можно распространять другим участникам;
- есть право обновлять `authorized_keys` для целевого пользователя.

### SID группы синхронизации

В одной сети может одновременно существовать несколько независимых групп хостов, которые не должны
обмениваться SSH-ключами друг с другом. Для разделения таких групп вводится SID — идентификатор
группы синхронизации.

SID должен передаваться во всех discovery-сообщениях, announcement и запросах обмена ключами.
Участник должен игнорировать сообщения и ключи с чужим SID, даже если отправитель находится в той же
локальной сети.

SID не является секретом и не заменяет аутентификацию. Он нужен для маршрутизации и логического
разделения групп.

Для безопасного обмена внутри группы вводится SID_TOKEN — секрет, связанный с конкретным SID.
SID_TOKEN используется для подписи discovery-сообщений, announcement и запросов обмена ключами.
Участники с одинаковым SID, но разным SID_TOKEN не должны принимать сообщения друг друга.

SID_TOKEN должен храниться как секрет конфигурации и не должен передаваться в открытом виде по сети,
попадать в логи или записываться в `authorized_keys`.

### Discovery

В первой версии discovery строится на двух источниках:

- статический список bootstrap-peers из конфигурации, переменной окружения или аргумента запуска;
- сетевые announcement-сообщения от участников с тем же SID.

Bootstrap-peers нужны, чтобы участник мог найти хотя бы часть группы без внешнего service discovery.
Это может быть список DNS-имен контейнеров, IP-адресов или service names внутри контейнерной сети.

Так как состав контейнеров может меняться, discovery не должен выполняться только один раз при
старте. Демон должен периодически перечитывать источник участников или получать уведомления от
механизма service discovery.

Дополнительно демон при запуске отправляет UDP broadcast announcement в локальную сеть: "я новый
участник, мой SID такой-то, мой адрес такой-то, мой порт такой-то". Уже работающие демоны с тем же
SID могут использовать такое объявление как сигнал немедленно запустить цикл синхронизации, не
дожидаясь следующего polling-интервала.

Announcement не должен сам по себе означать доверие. Получатель должен проверить HMAC-подпись
сообщения через SID_TOKEN. Announcement с неизвестным SID, чужим SID или некорректной подписью
должен быть проигнорирован.

Автоматическое обнаружение через Docker API, Kubernetes API или внешний service discovery можно
рассмотреть отдельно, когда базовый протокол обмена ключами будет стабилен.

### Режим демона

Базовый режим работы — долгоживущий процесс. Он выполняет начальную синхронизацию при старте, а
затем регулярно повторяет цикл:

1. обновить список известных участников;
2. обработать входящие объявления новых участников;
3. получить публичные ключи участников, которых еще нет локально;
4. опубликовать собственный публичный ключ для новых участников;
5. применить изменения к управляемому блоку `authorized_keys`.

Интервал синхронизации по умолчанию — 30 секунд с небольшим jitter, чтобы несколько контейнеров не
запускали обмен одновременно. Интервал должен быть настраиваемым. Polling дает надежную eventual
consistency, а announcement ускоряет подключение новых участников. Событийная модель через Docker,
Kubernetes или внешний discovery может быть добавлена позже.

Демон должен корректно обрабатывать завершение процесса: прекращать новые синхронизации, не
оставлять частично записанный `authorized_keys` и завершаться с понятным кодом.

### Обмен ключами

Минимальный поток одного цикла обмена:

1. локальный демон читает собственный публичный ключ;
2. локальный демон отправляет announcement при старте или изменении своего адреса/ключа;
3. локальный демон обращается к каждому известному или объявившемуся доверенному участнику;
4. участники возвращают свои публичные ключи;
5. локальный демон добавляет отсутствующие ключи в `authorized_keys`.

Обмен должен распространять только публичные ключи. Приватные ключи никогда не должны покидать
локальный хост.

Для первой версии обмен выполняется через простой HTTP API поверх TCP внутри контейнерной сети.
Каждый HTTP-запрос, HTTP-ответ и UDP announcement должен быть подписан HMAC-SHA256 через SID_TOKEN.
В подписываемые данные должны входить SID, идентификатор отправителя, timestamp, nonce и тело
сообщения. Это позволяет аутентифицировать участника, проверить целостность сообщения и не принимать
сообщения, переиспользованные между разными SID.

### Идемпотентность

Повторный запуск или повторный цикл демона не должен создавать дубликаты в `authorized_keys`.

Для этого инструмент должен:

- распознавать ключи, которыми он управляет;
- добавлять отсутствующие ключи;
- оставлять уже добавленные ключи без изменений;
- не удалять пользовательские ключи, добавленные вручную.

Практичный вариант — хранить управляемый блок в `authorized_keys` между маркерами, например:

```text
# ssh-key-sync begin
ssh-ed25519 AAAA... host-a
ssh-ed25519 BBBB... host-b
# ssh-key-sync end
```

Так инструмент сможет обновлять только свой участок файла и не затрагивать остальное содержимое.

### Безопасность

Безопасный обмен ключами является обязательным требованием, а не дополнительной возможностью.
Инструмент не должен принимать и устанавливать публичные ключи без проверки источника.

Так как публичный ключ дает право на SSH-доступ, инструмент должен защищаться от подмены участников.
Для первого варианта минимально необходимо:

- явно задавать список доверенных участников;
- не принимать ключи от неизвестных хостов;
- не принимать ключи от участников с другим SID;
- не принимать сообщения с некорректной подписью SID_TOKEN;
- не считать сетевое announcement достаточным основанием для доверия;
- аутентифицировать участника перед приемом его публичного ключа;
- проверять целостность и ожидаемый формат полученного ключа;
- логировать, какие ключи и от каких участников были добавлены;
- не передавать приватные ключи;
- выставлять корректные права на `~/.ssh` и `authorized_keys`.

Подлинность участника в первой версии подтверждается SID_TOKEN, который отличается от SID. SID не
является секретом, но участвует в подписываемых данных, чтобы сообщение нельзя было перенести из
одной группы синхронизации в другую. Более строгие механизмы, такие как mTLS, предварительно
известные fingerprints или SSH host key pinning, можно добавить позже.

### Ошибки и частичная доступность

Если часть участников недоступна при запуске или во время очередного цикла демона, инструмент не
должен ломать уже настроенный доступ к доступным участникам. Ошибки синхронизации по отдельным
хостам должны быть явно видны в логах и метриках состояния демона.

Демон должен завершаться с ошибкой только при локальных критичных проблемах: неверная конфигурация,
отсутствующий SID, отсутствующий SID_TOKEN, невозможность прочитать собственный публичный ключ или
невозможность безопасно обновить `authorized_keys`.

Ошибки отдельных участников обрабатываются в мягком режиме: недоступные или не прошедшие проверку
участники пропускаются в текущем цикле, но ошибка должна быть видна в логах и состоянии демона.

## Alternatives

### Ручное управление `authorized_keys`

Можно заранее собирать публичные ключи и монтировать готовый `authorized_keys` в каждый контейнер.
Это просто, но плохо подходит для динамических окружений и требует внешней автоматизации.

### Централизованное хранилище ключей

Можно хранить все публичные ключи в одном внешнем хранилище: Consul, etcd, S3, Kubernetes Secret или
другом сервисе. Это упрощает распространение ключей, но добавляет внешнюю зависимость и усложняет
запуск в небольших окружениях.

### Использование только SSH host keys

SSH host keys помогают проверить идентичность сервера, но не решают задачу авторизации пользователей
через `authorized_keys`. Поэтому они могут быть частью проверки безопасности, но не заменяют обмен
публичными пользовательскими ключами.

### Использование общей пары ключей

Можно выдать всем контейнерам один и тот же приватный ключ. Это проще, но небезопасно: компрометация
одного контейнера компрометирует доступ от имени всех участников. Такой вариант не подходит как
основной дизайн.

## Voting

Принятые решения для первой версии:

- discovery использует bootstrap-peers плюс сетевые announcement-сообщения;
- announcement отправляется в локальную сеть через UDP broadcast и фильтруется по SID;
- SID и SID_TOKEN задаются явно через конфигурацию, переменную окружения или аргумент запуска, без
  значений по умолчанию;
- SID не является секретом, но обязательно входит в подписываемые данные;
- обмен публичными ключами выполняется через HTTP API поверх TCP;
- безопасность обмена обеспечивается HMAC-SHA256 через SID_TOKEN;
- интервал фоновой синхронизации по умолчанию — 30 секунд с jitter;
- локальные критичные ошибки завершают демон, ошибки отдельных участников не останавливают демон;
- первая версия только добавляет и обновляет управляемый блок ключей, но не удаляет ключи исчезнувших
  участников автоматически.

## Migration

Для существующих окружений миграция может быть постепенной:

1. подготовить отдельную тестовую группу контейнеров;
2. запустить инструмент в режиме, который только показывает планируемые изменения `authorized_keys`;
3. проверить список участников и fingerprints ключей;
4. включить запись управляемого блока в `authorized_keys`;
5. после успешной проверки запускать инструмент как демон вместе с контейнером.

Существующие пользовательские записи в `authorized_keys` не должны удаляться или изменяться.
